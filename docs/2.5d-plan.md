# 2.5D 化迁移计划

## 背景

当前画面观感仍是 Flash 风（纯填色 + 硬边 + 单层 + 无光照），而引擎定位是 "2.5D 短剧生产底座"。问题不在引擎，在美术资产层 — procedural shape 语言已经具备做 2.5D 的全部表达力（线性/径向渐变、`transform` 嵌套、数值表达式、`when` 条件），但现有 sample 资产没有充分利用。同时已有的"2.5D 钩子"（场景的 `foreground` 字段、角色 y-sort、camera zoom）还停留在数据层，没在渲染管线里被消费。

这份文档列出推进到全面 2.5D 的范围、阶段、改动点和验收标准。

## 目标定义

**"2.5D" = 在 2D 图形系统里通过下列任一手段制造深度幻觉**：

1. **多层视差（parallax）** — 镜头横移时背景慢、中景中、前景快
2. **深度排序 + 遮挡（occlusion）** — 前景物件（茶几、栏杆、植物）能挡住角色一部分
3. **伪深度缩放（z-scaling）** — 角色站位有 `z` 值，远处自动缩小 + 降亮
4. **软光照（soft lighting）** — 角色身上有渐变光照、面部高光、下颌阴影
5. **大气透视（atmospheric perspective）** — 远处背景饱和度降、对比度降
6. **接触阴影（contact shadow）** — 软边、跟随尺度变化的脚下投影

到达验收点：客厅 demo 看上去更像「迪士尼分层风」/「Hollow Knight 风」，而不是「Flash MX 风」。

## 非目标

- 不引入 WebGL / Three.js — 全部继续基于 Canvas2D
- 不引入骨骼 / IK / 物理 — 动作仍走 `transform + sin(time)` 数值表达式
- 不引入 3D 网格 / 法线贴图
- 不引入资产编辑器 GUI（拖拽改 shape） — 仍走 manifest + AI 生成

## 范围拆解

按"现有示例转化 / AI Skill 升级 / 引擎实现"三轴拆，每轴独立可推进，最后汇合验收。

### 轴 A — 引擎实现（基础设施）

新加的内核能力。无 UI 改动，只动 [src/engine/proceduralShape.ts](../src/engine/proceduralShape.ts) 和 [src/components/PreviewCanvas.tsx](../src/components/PreviewCanvas.tsx)。

#### A1. Foreground 层渲染

- 目前 `SceneDefinition.foreground` 字段被忽略
- 新行为：在 character + effect 渲染**之后**，从 `scene.foreground` 查 asset，调 `paintShape` 画一遍
- 该层默认共享场景的 camera transform，自动获得遮挡角色的效果
- 影响文件：[PreviewCanvas.tsx](../src/components/PreviewCanvas.tsx)

#### A2. 角色 z 坐标 + 深度缩放

- schema 扩展：`characterAppear.position` 可选 `z?: number`，默认 0
- `CharacterRenderState` 加 `z: number`
- 渲染：`depthScale = 1 / (1 + z * 0.0015)`、`depthDim = 1 - clamp(z * 0.0008, 0, 0.25)`
- 把 `ctx.globalAlpha = 1 - depthDim` 应用到 `drawCharacter` 调用
- timeline 事件 `characterMove` 也加可选 `to.z`，移动时按 ease 插值

#### A3. 多层 Scene Shape

- ProceduralShape 新加可选字段 `layers?: { background?: Primitives; midground?: Primitives; foreground?: Primitives }`
- 向后兼容：原 `primitives` 字段视为 `midground`
- 渲染管线：在 `drawShape` 之外新加 `drawSceneLayers(ctx, asset, state, layer)`，分三次调用
- PreviewCanvas 渲染顺序：`bg → props → characters(y-sort) → effects → midground(暂无) → fg`

#### A4. 视差因子

- ProceduralShape.layers 每层可选 `parallaxFactor: number`（默认 1.0）
- 摄像机 transform 对每层用不同 factor：`ctx.translate(-camera.x * factor, -camera.y * factor)`
- background 用 0.4 ~ 0.6，midground 1.0，foreground 1.2 ~ 1.5
- camera 移动时立刻能看出深度

#### A5. Mask / Clip primitive（可选高级）

- 新原语 `clip: { kind: "clip"; shape: Primitive; children: Primitive[] }`
- 把 shape 作为 clipping region，children 只在区域内绘制
- 用途：让光照高光被限制在面部圆形内、衣服花纹被限制在身体形状内
- 实现：`ctx.save(); ctx.clip(); …children…; ctx.restore()`

#### A6. Soft shadow primitive 或 modifier

- 选项一：新原语 `softShadow: { kind, ...child原语, blur, offset, color }`
  - 渲染时给 ctx 设置 `shadowBlur`、`shadowColor`、`shadowOffsetX/Y`，再画子原语
- 选项二：在普通原语上加可选 `shadow?: { blur, offset, color }` 修饰符
- 选项二更轻量，推荐先做选项二

#### A7. 接触阴影自适应

- 角色脚下椭圆投影现在是固定大小
- 改成跟着 `character.scale * depthScale` 一起缩放、透明度跟着 z 提升而下降
- 通过 shape 表达式即可实现：`rx: "92 * (1 - clamp(z * 0.0008, 0, 0.3))"`
- 不需要引擎改动，但需要 A2 把 z 透传进 shape state

## 风险与权衡

**性能** — 多层 + 渐变 + radial gradient 让 Canvas2D draw call 翻倍。当前 sample 21 primitives × 2 角色 + 16 场景 ≈ 60 draws/帧。预计升级后 ≈ 200 draws/帧。Canvas2D 在 200~500 draws 范围依然 60fps，但需要关注：
- gradient 对象每帧重建开销（缓存策略：把 gradient 算好挂在 ctx state 上）
- 透明度叠加导致的 fill rate 压力

**复杂度** — shape 语言会从 9 个 primitives 涨到 ~15 个（加 clip / shadow / multi-layer）。AI 生成的失败率会上升，需要 reference 文档配套升级 + 加入 schema 校验。

**视觉一致性** — 多人手工迁移容易出现"父亲有渐变、孩子还是纯色"。建议加 lint 脚本：检测 character manifest 是否包含至少 1 个 gradient + 1 个 radial shadow，否则警告。

**摄像机系统** — parallax 要求 background layer 用不同的 transform。如果未来引入"镜头跟随角色"这种动态行为，要保证 parallax 仍正确（factor 用相对偏移而非绝对位置）。
